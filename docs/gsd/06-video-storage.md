## Goal
Описать модуль управления видео: как загружаются, хранятся и раздаются видеофайлы для лендингов, какие форматы и ограничения поддерживаются, и как генерируется HTML‑код видеоплеера.

## Solution
Видео загружаются через админский интерфейс `/admin/videos`, который использует API `/api/admin/videos/upload`. Файлы сохраняются в `public/video/*` или `public/uploads/*` (в зависимости от реализации), а доступ к ним осуществляется либо напрямую по URL (`/video/*.mp4`), либо через API‑роут `/api/video/[...path]`. Модуль хранит метаданные (имя файла, оригинальное имя, размер, путь, дата создания) в таблице `videos` (или JSON‑поле), а для вариантов товара в `Variant` используются поля `videoUrl*` и `videoHtml*` для встраивания видео в лендинг.

## Details

### Схема хранения и БД

Текущая реализация использует файловую систему `public/video` и `public/uploads`, а также поля в модели `Variant`:

```text
model Variant {
  ...
  videoUrl              String?
  videoUrlDesktop       String?
  videoHtmlDesktop      String?  @db.Text
  videoUrlMobile        String?
  videoHtmlMobile       String?  @db.Text
  videoTitle            String?
  videoTitleRu          String?
  videoText             String?  @db.Text
  videoTextRu           String?  @db.Text
  ...
}
```

Дополнительно можно выделить отдельную таблицу `Video` (если нужно более формальное хранение), со схемой:

```text
model Video {
  id           String   @id @default(uuid())
  filename     String
  originalName String
  size         Int
  mimeType     String
  path         String   // относительный путь в public/...
  createdAt    DateTime @default(now())

  // при необходимости: ссылки на Variant/Landing
}
```

Если таблицы `Video` нет, аналогичная информация может храниться в JSON‑структурах или извлекаться из имени файла.

### Поддерживаемые форматы и ограничения

По ТЗ:

- Форматы:
  - MP4
  - WebM
  - OGG
  - MOV
- Максимальный размер:
  - 100 МБ (проверяется на уровне загрузки, либо через конфиг сервера).

Формат должен поддерживаться браузерами целевой аудитории; предпочтительно использовать MP4 (H.264) как наиболее совместимый.

### API‑маршруты для видео

#### Admin‑маршруты

- **GET `/admin/videos`**
  - Страница админки, использующая внутренний API для списка видео.

- **POST `/api/admin/videos/upload`**
  - Назначение: загрузка нового видеофайла.
  - Поведение:
    - принимает multipart/form‑data с файлом;
    - проверяет размер и тип;
    - сохраняет файл в `public/video` (например, `public/video/{uuid}_raw_video.mp4`);
    - возвращает метаданные и URL, по которому видео будет доступно.

- **GET `/api/admin/videos`** (если реализовано)
  - список загруженных видео (для отображения в админке).

#### Публичные маршруты

- **GET `/video/*.mp4` / `/video/*`**
  - статическая раздача файлов из `public/video`.

- **GET `/api/video/[...path]`** (если реализовано)
  - более гибкий роут, позволяющий:
    - логировать просмотры,
    - проверять доступ,
    - реализовывать alias‑ы и т.д.

### Генерация HTML‑плеера

В варианте товара для встраивания видео используются поля:

- `videoHtmlDesktop`
- `videoHtmlMobile`

Типичный HTML для плеера:

```html
<video
  src="/video/kostyum-zimova-progulyanka-new-site-web.mp4"
  autoplay
  loop
  muted
  playsinline
  controls
>
  Ваш браузер не поддерживает HTML5-видео.
</video>
```

Рекомендуемые атрибуты:

- `autoplay` + `muted` + `playsinline` — для автозапуска без звука, особенно на мобильных.
- `loop` — зацикливание.
- `controls` — опционально, если нужен стандартный плеер.

Обычно HTML‑код генерируется утилитой или вручную вставляется в админке, после чего сохраняется в `videoHtmlDesktop` / `videoHtmlMobile` и просто рендерится на лендинге через `dangerouslySetInnerHTML`.

### Связь с вариантаи товара

- Вариант может ссылаться на одно или несколько видео:
  - через прямой URL (`videoUrl`, `videoUrlDesktop`, `videoUrlMobile`);
  - через HTML‑вставки (`videoHtmlDesktop`, `videoHtmlMobile`).
- Рендеринг на лендинге:
  - отдельные компоненты для видео‑блока читают эти поля и рендерят соответствующий плеер.
  - если поля пусты — блок не отображается.

### Edge cases и рекомендации

- **Размер и формат файла**:
  - при загрузке файла больше 100 МБ нужно явно отклонять запрос с понятным сообщением для администратора;
  - форматы, не попадающие в список (например, AVI, MKV) — либо запрещать, либо конвертировать до загрузки.
- **Кэширование и CDN**:
  - видеофайлы могут быть довольно большими, стоит использовать:
    - заголовки кэширования на уровне Nginx,
    - либо вынести `public/video` на CDN/object storage (S3‑совместимое хранилище).
- **Мобильные устройства и автоплей**:
  - многие мобильные браузеры блокируют автоплей со звуком, поэтому **обязателен** `muted` для автозапуска;
  - для мобильных лучше иметь облегчённую версию видео (меньшее разрешение/битрейт).
- **Безопасность HTML‑вставок**:
  - `videoHtmlDesktop/Mobile` — сырой HTML, поэтому:
    - админ‑панель должна доверять только администраторам,
    - в идеале применять санитайзер для HTML, чтобы исключить XSS.
- **Удаление видео**:
  - при удалении видеофайла нужно убедиться, что ни один вариант больше на него не ссылается;
  - можно реализовать soft‑delete или пометку «не использовать в новых вариантах».
