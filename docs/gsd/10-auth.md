## Goal
Описать модуль аутентификации: как защищается Admin UI от несанкционированного доступа, как устроены маршруты `/api/auth/*`, таблица пользователей и middleware, а также какие роли предусмотрены.

## Solution
Админ‑панель `/admin/*` защищена с помощью JWT‑аутентификации, где токен хранится в httpOnly‑cookie. Авторизация реализована через API‑маршруты `/api/auth/login`, `/api/auth/logout`, `/api/auth/me`. В БД есть таблица `User`, хранящая администраторов (email + хэш пароля). Middleware/утилита `verifyAuth` проверяет токен и блокирует доступ к защищённым API‑маршрутам.

## Details

### Схема БД: таблица `User`

Фрагмент `schema.prisma`:

```text
model User {
  id           String   @id @default(uuid())
  email        String   @unique
  passwordHash String
  createdAt    DateTime @default(now())
}
```

На данный момент система предполагает одну роль — администратор; дальнейшая роль‑based модель может быть расширена.

### Маршруты `/api/auth/*`

Реализация находится в `app/api/auth/*`.

- **POST `/api/auth/login`**
  - Вход:
    - `email`
    - `password`
  - Логика:
    - ищет пользователя по `email` в таблице `User`;
    - сравнивает переданный пароль с `passwordHash` (через `bcryptjs`);
    - при успехе:
      - создаёт JWT с данными пользователя (id, email),
      - записывает его в httpOnly‑cookie;
      - возвращает `{ success: true }`.
    - при неуспехе — HTTP 401.

- **POST `/api/auth/logout`**
  - Удаляет/очищает auth‑cookie.
  - Возвращает `{ success: true }`.

- **GET `/api/auth/me`**
  - Проверяет токен в cookie.
  - При валидном токене возвращает информацию о текущем пользователе (id, email).
  - При отсутствии/некорректности токена — HTTP 401.

### Middleware / защита admin‑маршрутов

Для защиты admin‑API используется функция `verifyAuth` (`lib/auth.ts`):

- читается токен из httpOnly‑cookie;
- валидируется с помощью `JWT_SECRET`/`NEXTAUTH_SECRET`;
- если токен невалиден:
  - бросается ошибка `Unauthorized`, которая в обработчике переводится в HTTP 401;
- если валиден:
  - в контексте запроса считаем пользователя аутентифицированным.

Большинство маршрутов под `/api/admin/*` начинается с вызова `await verifyAuth()`.

### Admin UI

- Страницы `/admin/login`:
  - форма логина с полями email/пароль,
  - при отправке вызывает `POST /api/auth/login`,
  - при успехе перенаправляет на `/admin/landings`.
- Обёртка/guard `auth-check`:
  - на клиенте и/или сервере проверяет, что пользователь авторизован;
  - при отсутствии сессии редиректит на `/admin/login`.

### ENV‑переменные

- `ADMIN_EMAIL`, `ADMIN_PASSWORD` — начальные учётные данные, используемые в сидинге.
- `NEXTAUTH_SECRET` / `JWT_SECRET` — секрет для подписи JWT.
- `NEXTAUTH_URL` — базовый URL приложения.

### Edge cases и рекомендации

- **Безопасность cookie**:
  - флаги:
    - `httpOnly` — обязательно;
    - `secure` — включать в production;
    - `sameSite` — как минимум `lax`.
- **Сброс пароля**:
  - в текущей версии нет полноценного UI для смены/сброса пароля;
  - смена осуществляется либо через сид‑скрипт, либо напрямую в БД.
- **Мультиадмин**:
  - таблица `User` поддерживает несколько записей, но UI управления пользователями может отсутствовать;
  - при необходимости можно добавить модуль управления пользователями на `/admin/users`.
- **Роли**:
  - сейчас фактически одна роль — администратор;
  - для разделения прав (маркетолог/контентщик/владелец) потребуется добавить поле `role` в `User` и авторизационную логику в `verifyAuth` и UI.
